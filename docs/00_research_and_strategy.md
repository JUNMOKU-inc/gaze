# Gaze — LLMインプット特化スクリーンキャプチャツール
## コンセプト・事業戦略・市場調査レポート

> 作成日: 2026-03-25
> ステータス: 初期コンセプト

---

## Executive Summary

**一言で言うと**: 「スクリーンショットを撮る → LLMに最適な形で即座に渡す」を1キーストロークで実現するMacネイティブアプリ。

LLM時代において、スクリーンショットはもはや「人に見せるもの」ではなく「AIに理解させるもの」に変わりつつある。しかし現在のスクリーンショットツール（CleanShot X、Shottr等）はすべて人間向けに最適化されており、LLMへのインプットワークフローは完全に手動で断片的。このギャップを埋める専用ツールに大きな市場機会がある。

## 1. 市場調査サマリー

### 1.1 競合環境

| ツール | 価格 | スクショ | 録画 | LLM連携 | ポジション |
|--------|------|---------|------|---------|-----------|
| CleanShot X | $29 + $19/yr | ◎ | ◎ | ✗ | 市場リーダー（汎用） |
| Shottr | $8-30 | ◎ | ✗ | ✗ | 開発者向け無料代替 |
| Xnapper | $30-80 | ○ | ✗ | ✗ | SNS映え特化 |
| Snagit | $63 | ◎ | ◎ | △ | エンタープライズ |
| Loom | $0-24/mo | ✗ | ◎ | △ | 非同期コミュニケーション |
| **LazyScreenshots** | **$29** | **○** | **✗** | **◎** | **AI開発者向け（直接競合）** |
| ScreenSnapAI | $0-20 | ○ | ✗ | ○ | AI整理特化 |
| Claude Desktop内蔵 | 無料 | △ | ✗ | ○ | Claude限定 |

**重要な発見**:
- LazyScreenshots（$29）が唯一の本格的競合。バーストモード・AI自動ペースト機能を持つ
- しかし**動画対応なし**、**LLM最適化（リサイズ・圧縮）なし**、**マルチプロバイダ対応なし**
- 伝統的スクショツール（CleanShot X等）はAI/LLM連携が**ゼロ**

### 1.2 LLMの画像入力の現状

| プロバイダ | 対応形式 | 最大サイズ | 最適解像度 | トークンコスト（1000x1000px） |
|-----------|---------|-----------|-----------|--------------------------|
| Claude | JPEG, PNG, GIF, WebP | 5MB (API) | 1568px長辺 | ~1,334トークン (~$0.004) |
| GPT-4o | JPEG, PNG, WebP, GIF | 50MB | 2048px/512pxタイル | ~765トークン (high) |
| Gemini | PNG, JPEG, WebP, HEIC | 20MB | 768pxタイル | ~1,290トークン |

**動画サポート**: Geminiのみネイティブ対応（1FPS、258トークン/秒）。Claude・GPT-4oはフレーム抽出が必要。

### 1.3 ユーザーペインポイント

調査から特定された主要な課題:

1. **ワークフローの断絶**: キャプチャ → クロップ → アノテーション → 保存 → アップロード → プロンプト記述、が全て手動
2. **不要な情報の混入**: ツールバー、通知、ステータスバーなどのUIクロームがLLMのコンテキストを汚染
3. **解像度の不一致**: 高解像度Retinaスクリーンの画像がLLM側で自動圧縮され、テキストが読めなくなる
4. **フォーマット変換**: HEIC→JPEG/PNG変換が毎回必要
5. **複数画像のコンテキスト**: Before/After比較やマルチステップの操作手順を伝えるのが困難
6. **トークンコストの無駄**: 不必要に大きな画像を送信し、トークンを浪費
7. **数値の誤認識**: LLMは画像内の数字を頻繁に誤読する

## 2. プロダクトコンセプト

### 2.1 ビジョン

> **「Capture for AI」—— AIに渡すための、世界最速のスクリーンキャプチャ**

従来のスクショツールが「人間が見る画像」を作るのに対し、本製品は「AIが最も効率的に理解できる入力」を生成することに特化する。

### 2.2 コア・バリュー・プロポジション

```
1キーストローク → キャプチャ → AI最適化 → クリップボード → LLMにペースト
```

**差別化の3本柱**:

1. **AI-Optimized Output**: 各LLMプロバイダの仕様に合わせた自動リサイズ・圧縮・フォーマット変換
2. **Context-Aware Capture**: 不要なUI要素の自動除去、関連部分のスマートクロップ
3. **Zero-Friction Workflow**: キャプチャからLLMへのペーストまでを最小ステップで完結

### 2.3 ターゲットユーザー

**Primary**: AIを日常的に使う開発者・デザイナー・ナレッジワーカー
- Claude、ChatGPT、Cursor、Copilot等を毎日使用
- スクショを頻繁にLLMに送信してコードレビュー、UIフィードバック、ドキュメント分析を行う
- Mac使用、生産性ツールへの投資に積極的

**Secondary**: テクニカルライター、QAエンジニア、プロダクトマネージャー
- バグレポートやドキュメント作成でスクショを多用
- AIを活用した効率化に関心

### 2.4 MVP機能（v1.0）

#### Must Have（リリース必須）

| 機能 | 説明 | 差別化ポイント |

| **エリアキャプチャ** | 画面の任意の範囲を選択して撮影 | ベースライン |

## 3. 技術戦略

### 3.1 技術スタック

**Swift + SwiftUI（ネイティブMac）** を採用する理由:

| 要件 | Swift/SwiftUI | Electron | Tauri |
|------|-------------|----------|-------|
| ScreenCaptureKit直接アクセス | ◎ | ✗ | △（FFI経由） |
| アプリサイズ | 5-15MB | 100-300MB | 10-25MB |
| メモリ使用量 | 最小 | 大 | 小 |
| macOS統合（メニューバー、ホットキー） | ◎ | △ | ○ |
| ネイティブUX | ◎ | ✗ | △ |
| 開発速度 | ○ | ◎ | ○ |

**結論**: スクリーンキャプチャツールはOS深層との統合が不可欠。Swift/SwiftUI一択。

### 3.2 コアアーキテクチャ

```
┌─────────────────────────────────────────────┐
│                  Gaze                   │
├─────────────────────────────────────────────┤
│  Menu Bar App (SwiftUI)                      │

│  │   ├── Area Capture                        │
│  │   ├── Window Capture                      │
│  │   ├── Scrolling Capture                   │
│  │   └── Video Capture                       │
│  ├── Image Pipeline                          │
│  │   ├── Smart Crop (Vision framework)       │

```

### 3.3 必要なシステム権限

| 権限 | 用途 | 取得タイミング |
|------|------|-------------|

**注意**: macOS Sequoiaから画面録画権限のプロンプトが毎週表示されるようになった。UXで対処が必要。

### 3.4 対応OS

- **最小**: macOS 14 (Sonoma) — ScreenCaptureKit の最新機能を活用
- **推奨**: macOS 15 (Sequoia) 以降

## 5. 競合優位性と参入障壁

### 5.1 vs CleanShot X

| 観点 | CleanShot X | Gaze |
|------|------------|-----------|
| スクショ機能 | ◎ 50+機能 | ○ 必要十分 |
| 動画機能 | ◎ | ○ |
| LLM最適化 | ✗ なし | ◎ コア機能 |
| 価格 | $29 + $19/yr | $19（初年度込み） |
| ポジション | 汎用スクショ | AI特化スクショ |

**勝ち筋**: CleanShot Xと全機能で競わない。「LLMに送るならGaze」という明確なポジションを取る。

### 5.2 vs LazyScreenshots（最も近い直接競合）

| 観点 | LazyScreenshots | Gaze |
|------|----------------|-----------|
| AI自動ペースト | ◎ | ◎ |
| バーストモード | ◎ | ◎ |
| LLMプロバイダ別最適化 | ✗ | ◎ |
| トークンコスト表示 | ✗ | ◎ |
| 動画 → フレーム抽出 | ✗ | ◎ |
| OCR抽出モード | ✗ | ◎ |
| スマートクロップ | ✗ | ◎ |
| 価格 | $29 | $19 |

**勝ち筋**: LazyScreenshotsはキャプチャ→ペーストの自動化に留まる。GazeはLLMが最も効率的に理解できる形への最適化まで踏み込む。

### 5.3 長期的な参入障壁

1. **LLM仕様の継続追従**: 各プロバイダのビジョンAPI仕様は頻繁に変わる。最新仕様への即時対応が競争力
2. **最適化ノウハウの蓄積**: どの解像度・フォーマット・アノテーションがLLMの理解度を最大化するかのデータ蓄積
3. **ネットワーク効果**: ユーザーコミュニティからのフィードバックによるプロダクト改善サイクル
4. **Apple Foundation Models統合**: macOS上のオンデバイスAIとの統合で独自機能を構築

## Appendix A: LLM画像入力仕様詳細

### Claude (Anthropic)
- 対応形式: JPEG, PNG, GIF, WebP
- 最大サイズ: 5MB (API), 10MB (claude.ai)
- 最大解像度: 8000x8000px（それ以上は拒否）
- 自動ダウンスケール: 長辺 >1568px の場合
- トークン計算: `tokens = (width × height) / 750`
- 最適: 1.15メガピクセル以下（1568×1568px）

### GPT-4o (OpenAI)
- 対応形式: JPEG, PNG, WebP, 非アニメーションGIF
- 2モード: `low`（常に85トークン）、`high`（512pxタイル、170トークン/タイル + 85）
- 最大: 2048×2048px にスケール後タイル化

### Gemini (Google)
- 対応形式: PNG, JPEG, WebP, HEIC, HEIF
- 小画像(≤384px): 258トークン固定
- 大画像: 768pxタイル、258トークン/タイル
- 動画: ネイティブ対応、1FPS、258トークン/秒
- `media_resolution`パラメータで解像度制御可能
