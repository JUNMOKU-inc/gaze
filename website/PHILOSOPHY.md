# Gaze LP — 設計思想

## コアメッセージ

**「AIに画面を見せよう。」**

Gazeは「スクリーンショットツール」ではない。AIに画面を見せるためのインターフェース。

## 思想の変遷

このLPは複数回のピボットを経て、現在の形にたどり着いた。

### v1: LLM画像最適化ツール（破棄）

最初の訴求は「LLMプロバイダ別に画像サイズを最適化してトークンコストを削減」だった。

破棄した理由:

- Claude/GPT-4o/Gemini全てが画像の自動リサイズを実装済み
- 画像1枚あたり$0.002。30%削減しても$0.001。体感不可能
- Claude DesktopとChatGPTがスクショ機能を内蔵し、「撮って貼る」を無料で代替

教訓: 技術的に正しいことと、ユーザーにとって価値があることは別。

### v2: OCR + MCP + バースト（改善したが不十分）

次に訴求を「OCRテキスト化」「MCPサーバー」「バーストモード」に切り替えた。機能としては正しいが、LP全体が「機能の羅列」に終始していた。

### v3: CLI-first（方向性は正しいが、伝え方がAI臭い）

CLIがMCPよりトークン効率10-32倍良く、信頼性100%であることがベンチマークで判明。CLI-firstに転換。しかしコピーが「AIが書いたマーケティング文」のまま。

### v4: CLIツールのLP作法（改善）

gogcli.sh、Charm、ripgrep等のCLIツールLPを調査。共通パターンを発見:

- 最初の1行で「何であるか」が完結
- コマンド例で語る。説明するな、見せろ
- 「革命的」「画期的」は使わない
- ユーザーの現状を否定しない

### v5: 現在 —「AIに画面を見せよう」

ユーザーとの対話から生まれた核心的な気づき:

> 今まさにアプリを開発するために、表示位置などをAIに教えたくて、何枚ものスクショを撮ってはAIに添付している。それをAI自身ができるようになる。

「ツールの機能」ではなく「AIの新しい能力」の話。これがヘッドラインになった。

## 設計原則

### 1. 見せる → 判断する → 動く

Gazeの価値は「スクショを撮る」で終わらない。

```
AIが全画面を見る → 気になる箇所を特定 → 座標指定で拡大 → テキスト化して読む → 次の処理へ
```

この自律的なループが回ることで、今まで人間が手でやっていた作業をAIが自動で完結できる。LPはこの「ループ」を伝える構成にしている。

### 2. 機能ではなくユースケース

Features セクションのカードは機能名（「OCRテキスト化」「バーストモード」）ではなく、ユースケース（「UIをAIに見せて直してもらう」「エラー画面をそのまま伝える」）で書く。

ユーザーは機能を買わない。自分の問題が解決されることを買う。

### 3. コマンド例で語る

CLIツールは、コマンド例が最も雄弁な説明文。

```bash
# AIに画面全体を見せる
$ gaze capture

# 気になる部分だけ拡大して見せる
$ gaze capture --area 340,120,800,500

# 画像じゃなくテキストで見せる
$ gaze ocr --area 340,120,800,500
```

この3行で、「何ができるか」が完全に伝わる。それ以上の説明は不要。

### 4. 書かないことを決める

- ❌ 「革命的」「画期的」「最速」「最強」
- ❌ ユーザーの現状を否定する表現（「その手間、終わりにしよう」）
- ❌ 「簡単」「シンプル」（コマンド1行を見せれば伝わる）
- ❌ 根拠のない数字（「毎週何時間も節約」）
- ❌ PII、DPA等の専門略語（「機密情報」と書く）
- ❌ 感嘆符

### 5. CLIツールのLP作法に従う

gogcli.sh、ripgrep、zoxide等から学んだパターン:

- **最初の1行**: `[何であるか]` が完結する（`AIに画面を見せよう。`）
- **ターミナルデモ**: インストールから使用までのストーリーを見せる
- **Value Props**: 3-4枚のカードに事実だけ。形容詞は最小限
- **比較級1語**: `smarter`, `faster` 程度。煽らない

### 6. 「見せる」を軸にした構成

Value Propsの4枚は全て「見せる」の変奏:

| 方法             | 説明                              |
| ---------------- | --------------------------------- |
| 画像で見せる     | スクショ。全画面もエリア指定も    |
| テキストで見せる | OCR。画像の1/20のトークンで伝わる |
| 動画で見せる     | 録画。操作の流れを丸ごと          |
| AIから直接見る   | CLI経由でAIが自分でキャプチャ     |

## アーキテクチャ思想

**CLI-first, MCP-also, GUI-too**

1. **CLI** — AIエージェント・自動化向け。最優先。`brew install gaze`
2. **MCP** — IDE統合向け。CLIの薄ラッパー
3. **GUI** — 人間向け。Tauriアプリ

CLIが本体。MCPとGUIはCLIの上に乗るインターフェース。

## 競合との位置づけ

CleanShot X等の汎用スクショツールとは比較しない。カテゴリが違う。

直接競合はPeekaboo（macOS CLIキャプチャ + MCP）。差別化:

- Peekabooは汎用macOS自動化（click, type, scroll）
- Gazeは「AIに画面を見せる」に特化（OCR、バースト、機密情報マスク、パイプライン）

## ターゲット

AIと毎日会話している開発者。Claude Code、Cursor、ChatGPTを使っていて、何枚ものスクショを撮ってはAIに貼り付けている人。

その人に「AIが自分で画面を見に行けるようになる」と伝えれば、価値は一瞬で伝わる。
